# Visão Geral

Até agora, focamos em modificar o **código** dos jogos. Mas o código é apenas uma pequena parte do que compõe um jogo. Veja este cenário do Urban Terror:

![Urban Terror](/assets/images/8/1/urbanterror.jpg)

Para renderizar esse cenário, o jogo precisa carregar:
- **Modelos 3D** (jogadores, armas, mapa)
- **Texturas** (roupas dos jogadores, asfalto)
- **Animações** (flash do disparo)
- **Sons** (som da arma)

# Recursos de Jogos

Modelos 3D, texturas, imagens, sons e dados gerais são chamados de **recursos** (assets). Ao contrário do código, recursos são **grandes em tamanho** e frequentemente:

- **Comprimidos** para reduzir espaço em disco
- **Empacotados** em arquivos customizados para dificultar o acesso

Para modificar recursos de um jogo, precisamos:
1. Identificar onde os recursos são armazenados
2. Entender como desempacotar esses arquivos
3. Modificar os recursos
4. Reempacotar no formato original

# Sistema de Arquivos

Para que os recursos persistam entre reinicializações do computador, eles precisam ser armazenados no **sistema de arquivos** (HDD/SSD). O computador carrega os recursos necessários na **RAM** durante o jogo (geralmente nas telas de carregamento) e os descarrega quando não são mais necessários.

Para acessar o sistema de arquivos, jogos usam APIs do Windows como `CreateFile`. Monitorando essas chamadas, podemos descobrir onde o jogo armazena seus arquivos.

# Save Data

Muitos jogos permitem salvar o progresso. Os dados de save precisam ser armazenados no sistema de arquivos para persistir entre sessões. Cada jogo implementa salvamento de forma diferente:

| Tipo | Exemplo |
|---|---|
| Texto simples | Level atual em `.txt` |
| Binário | Posição de todas as entidades em formato customizado |
| Banco de dados | SQLite com estado completo do mundo |
